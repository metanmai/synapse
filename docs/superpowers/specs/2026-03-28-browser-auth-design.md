# Browser-Based CLI Authentication

**Date:** 2026-03-28
**Status:** Approved
**Scope:** CLI (`mcp/src/`), Backend (`backend/src/api/auth.ts`), Frontend (new `/cli-auth` page)

## Summary

Add a browser-based login flow to the CLI, similar to Claude Code. The CLI opens the user's browser to the Synapse web app, the user logs in with any method (password, magic link, Google, GitHub), and the web app redirects back to the CLI with an API key. Uses PKCE-style security to prevent interception.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Auth surface | Web app handles all auth, CLI is just a launcher | Users get all auth methods (OAuth, magic links, etc.) without reimplementing in the terminal |
| Callback mechanism | Local HTTP server on localhost | Instant, proven pattern (Claude Code, GitHub CLI, Stripe CLI) |
| Security | PKCE-style code_verifier/code_challenge | Prevents local process interception — API key never travels over localhost |
| Wizard options | "Sign in with browser" (default) + "Paste an API key" | Signup also goes through browser — simplifies wizard to 2 options |
| Email+password | Removed from interactive wizard, kept via `login --email --password` flags | Browser is the primary path; flags serve CI/scripted use |

## Security Model

### PKCE Flow

```
CLI                          Browser                      Backend
 │                              │                            │
 ├─ generate code_verifier      │                            │
 ├─ hash → code_challenge       │                            │
 ├─ generate state              │                            │
 ├─ start localhost server      │                            │
 ├─ open browser ──────────────►│                            │
 │   /cli-auth?challenge=X      │                            │
 │   &state=S&port=P            │                            │
 │                              ├─ user logs in ────────────►│
 │                              │                            ├─ create API key
 │                              │                            ├─ store {challenge, api_key, email}
 │                              │◄── redirect with code ─────┤
 │◄── GET /callback?code=C ─────┤                            │
 │    &state=S                  │                            │
 ├─ validate state              │                            │
 ├─ POST /auth/cli-exchange ───────────────────────────────►│
 │   {code, code_verifier}      │                            ├─ verify SHA256(verifier)==challenge
 │                              │                            ├─ return {api_key, email}
 │◄─────────────────────────── {api_key, email} ────────────┤
 ├─ delete session              │                            ├─ delete session
 ├─ shutdown server             │                            │
 └─ continue with init          │                            │
```

### Threat Mitigations

| Threat | Mitigation |
|--------|------------|
| Local process submits fake callback | PKCE: callback only carries opaque `code`, not API key. Exchange requires `code_verifier` which never leaves CLI memory. |
| Network intercepts localhost callback | Callback carries only `code`. The exchange (`code` + `code_verifier` → API key) happens over HTTPS to the backend. |
| Reuse of callback URL | `code` is single-use — backend deletes after first exchange. Local server shuts down immediately. |
| CSRF / crafted callback URL | `state` parameter is random, generated by CLI, validated on callback. Attacker can't guess it. |
| Abandoned login flow | 120s CLI timeout. Backend expires pending sessions after 5 minutes. |

## CLI Changes

### Wizard Flow (wizard.ts)

The auth step simplifies to 2 options:

```
◆ Synapse setup
│
● How do you want to connect?
│
│  ● Sign in with browser  — opens synapsesync.app
│  ○ Paste an API key      — from the dashboard
```

"Sign in with browser" calls `browserAuth()` → glyph spinner "Waiting for browser login..." → on callback, continues to editor selection.

### Standalone `login` Subcommand

- `synapsesync-mcp login` (interactive, no flags) → uses browser auth as default
- `synapsesync-mcp login --email <e> --password <p>` → still works for CI/scripted use (unchanged)

### New File: `mcp/src/cli/browser-auth.ts`

Exports: `browserAuth(): Promise<{ api_key: string; email: string }>`

Implementation:
- Generates `code_verifier` (64 bytes, `crypto.randomBytes`) and `code_challenge` (SHA-256 hex)
- Generates `state` (`crypto.randomUUID()`)
- Starts HTTP server on `127.0.0.1:0` (OS picks port) using `node:http`
- Opens browser using `node:child_process` exec: `open` (macOS), `xdg-open` (Linux), `start` (Windows)
- URL: `https://synapsesync.app/cli-auth?challenge=<code_challenge>&state=<state>&port=<port>`
- Callback handler (`GET /callback`):
  - Validates `state` matches
  - Extracts `code` from query params
  - Serves HTML success page: "Login successful! You can close this tab." (coffee palette styled)
  - Calls `POST https://api.synapsesync.app/auth/cli-exchange` with `{ code, code_verifier }`
  - Resolves promise with `{ api_key, email }`
  - Shuts down server
- Timeout: 120 seconds → rejects with error, shuts down server
- Server binds to `127.0.0.1` only (not `0.0.0.0`)

## Backend Changes

Two new endpoints in `backend/src/api/auth.ts`:

### `POST /auth/cli-session`

Called by the frontend after the user logs in on the `/cli-auth` page.

- Input: `{ code_challenge: string }` (request must be authenticated — user's Supabase JWT)
- Creates API key for the user (label: "cli", replaces existing "cli" key)
- Generates single-use `code` (`crypto.randomUUID()`)
- Stores session: `{ code, code_challenge, api_key, email, created_at }` in in-memory Map with 5-minute TTL
- Returns: `{ code }`

### `POST /auth/cli-exchange`

Called by the CLI to exchange the authorization code for an API key.

- Input: `{ code: string, code_verifier: string }`
- No auth required (the PKCE exchange is the auth)
- Looks up session by `code`
- Computes `SHA256(code_verifier)` and compares to stored `code_challenge`
- If valid: returns `{ api_key, email }`, deletes session
- If invalid: returns 401
- If not found / expired: returns 404

### Session Storage

In-memory Map with TTL cleanup, same pattern as the existing rate limiter. Sessions expire after 5 minutes. Cleanup runs on each new session creation (evicts expired entries).

## Frontend Changes

### New Page: `frontend/src/routes/cli-auth/+page.svelte`

1. Reads `code_challenge`, `state`, `port` from URL query params
2. If missing params → shows error: "This page should be opened from the CLI"
3. If user already logged in (Supabase session) → skips to step 5
4. If not logged in → shows login UI (reuse existing auth components — password, magic link, Google, GitHub)
5. After auth succeeds → calls `POST /auth/cli-session` with `{ code_challenge }` (authenticated with Supabase JWT)
6. Gets back `{ code }`
7. Redirects to `http://localhost:<port>/callback?code=<code>&state=<state>`
8. Brief "Redirecting you back to the terminal..." message shown before redirect

### Edge Cases

- Missing query params → error message, no redirect
- User closes tab → CLI times out after 120s
- Port unreachable on redirect → browser shows connection error, user re-runs CLI
