# Capture + Distill: Automatic Session Capture & Knowledge Extraction

**Date:** 2026-04-02
**Status:** Design approved
**Depends on:** Existing Synapse write API, MCP server (`synapsesync-mcp`)

---

## Overview

Add automatic session capture from AI coding tools and LLM-powered knowledge extraction to Synapse. Sessions are captured silently via a background daemon with tool-specific adapters, then optionally distilled into structured knowledge (decisions, architecture notes, learnings) and written to the workspace through the existing Synapse write API.

**Design principles:**
- **Adapter pattern** -- tool-agnostic; adding a tool = one new file
- **Single responsibility** -- watcher watches, adapter parses, distiller distills, writer writes
- **Unix pipes** -- each stage is independent, composable, replaceable
- **Existing ecosystem untouched** -- no changes to API, database, dashboard, or MCP tools

**What changes in existing Synapse:** Nothing. The write API, workspace, dashboard, search, MCP tools -- all unchanged. Capture and distill are new components that feed into the existing system through the existing write endpoint.

## Standard Session Format

The common contract between capture and everything downstream. All adapters output this format. The distiller and writer never need to know which tool the session came from.

```typescript
interface CapturedSession {
  /** Unique session ID (generated by adapter) */
  id: string;
  /** Which tool produced this session */
  tool: "claude-code" | "cursor" | "codex" | "gemini";
  /** Absolute path to the project/workspace */
  projectPath: string;
  /** When the session started */
  startedAt: string; // ISO 8601
  /** When last updated */
  updatedAt: string;
  /** The conversation turns */
  messages: SessionMessage[];
}

interface SessionMessage {
  role: "user" | "assistant";
  content: string;
  timestamp: string; // ISO 8601
  /** Tool calls made by the assistant, if any */
  toolCalls?: { name: string; input?: string; output?: string }[];
}
```

## Adapter Interface

Each adapter implements one interface. Three methods, single responsibility. The adapter doesn't watch, doesn't store, doesn't decide what to do with the session. It answers: "given this file, here's the session."

```typescript
interface ToolAdapter {
  /** Which tool this adapter handles */
  tool: string;
  /** Filesystem paths to watch */
  watchPaths(): string[];
  /** Parse a filesystem event into a session, or null if irrelevant */
  parse(filePath: string): CapturedSession | null;
}
```

### Adapter Implementations

#### Claude Code Adapter

- **Watch path:** `~/.claude/projects/`
- **File format:** JSONL (append-only)
- **Strategy:** Track byte offset per file, read only new lines on each event. Map `display` field to user messages. Session boundaries determined by `sessionId` field.
- **Alternative path:** Claude Code hooks can trigger `npx synapsesync-mcp capture sync` at session end for more reliable, event-driven capture.
- **Gotcha:** `history.jsonl` only stores user prompts, not assistant responses. The `projects/` directory contains richer per-session JSONL with full conversation data.

#### Cursor Adapter

- **Watch path:** `~/Library/Application Support/Cursor/User/workspaceStorage/*/chatSessions/*.json`
- **File format:** JSON (full rewrite on every update)
- **Strategy:** Hash-compare on change events to detect actual content changes. Skip if file is locked (Cursor running -- handle gracefully). Parse `requests` array, map `message.text` to user, `response[].value` to assistant.
- **Gotcha:** Files are rewritten entirely, not appended. Use content hash to avoid reprocessing unchanged sessions. The SQLite `state.vscdb` is an alternative source but is typically locked during use.

#### Codex CLI Adapter

- **Watch path:** `~/.codex/sessions/` (recursive)
- **File format:** JSONL (append-only, organized in `YYYY/MM/DD/rollout-*.jsonl`)
- **Strategy:** Watch for new `rollout-*.jsonl` files via directory creation events. Track byte offsets on existing files. New date directories are created daily -- watcher must handle new subdirectory creation.
- **Gotcha:** Files may be compacted (rewritten) when exceeding `history.max_bytes`. Detect compaction by checking if file size decreased.

#### Gemini CLI Adapter

- **Watch path:** `~/.gemini/tmp/*/chats/*.json`
- **File format:** JSON (full rewrite on update)
- **Strategy:** Hash-compare on change events. Map role `"model"` to `"assistant"`. Parse `messages` array directly -- structure is closest to the standard format.
- **Gotcha:** Sessions organized by project hash. Files are fully rewritten, not appended. 30-day default retention means old files may disappear.

## Watcher

Listens for filesystem events, routes to the correct adapter.

```
filesystem event (file changed/created)
       |
       v
  which adapter owns this path?
       |
       v
  adapter.parse(filePath)
       |
       v
  CapturedSession (or null, skip)
       |
       v
  emit to output (callback / event)
```

- Uses `chokidar` for cross-platform filesystem watching (macOS FSEvents, Linux inotify)
- Maintains a registry of adapters and their watch paths
- Routes events to the matching adapter based on path prefix
- Emits parsed sessions to registered listeners
- Does not parse, store, or process sessions -- only routes

## Daemon

Keeps the watcher alive. Handles lifecycle and nothing else.

**CLI commands:**

```bash
npx synapsesync-mcp capture start      # start daemon (detached process)
npx synapsesync-mcp capture stop       # stop daemon
npx synapsesync-mcp capture restart    # restart daemon
npx synapsesync-mcp capture status     # health check (running/stopped, uptime, adapters)
npx synapsesync-mcp capture list       # show recently captured sessions
```

**Implementation:**
- Detached child process (same pattern as `eslint --daemon`, `tsc --watch`)
- PID file at `~/.synapse/capture.pid`
- Log file at `~/.synapse/capture.log`
- Config at `~/.synapse/capture.json` (which adapters enabled, distill settings)
- Ships as part of the existing `synapsesync-mcp` package -- no separate install

**Config example:**

```json
{
  "adapters": {
    "claude-code": { "enabled": true },
    "cursor": { "enabled": true },
    "codex": { "enabled": true },
    "gemini": { "enabled": true }
  },
  "distill": {
    "auto": false,
    "provider": "anthropic",
    "model": "claude-sonnet-4-6",
    "apiKey": null
  }
}
```

## Distiller

Separate stage. Reads a `CapturedSession`, calls an LLM, outputs markdown files written through the existing Synapse write API.

**Input:** `CapturedSession`
**Output:** Markdown files with workspace paths (`decisions/`, `architecture/`, `learnings/`)

**Process:**

1. Receive a captured session (from daemon pipeline, manual invocation, or hook)
2. Build a prompt instructing the LLM to extract:
   - **Decisions** -- choices made and why (-> `decisions/<topic>.md`)
   - **Architecture** -- system design, data flow, component descriptions (-> `architecture/<topic>.md`)
   - **Learnings** -- gotchas, debugging insights, things that surprised (-> `learnings/<topic>.md`)
3. Call the LLM with the session transcript
4. Parse the structured response
5. Write each extracted file via the existing Synapse `write` API
6. The distiller does not capture, does not watch, does not manage lifecycle

**CLI commands:**

```bash
npx synapsesync-mcp distill <session-id>   # distill a specific session
npx synapsesync-mcp distill --latest        # distill the most recent session
```

**Trigger modes:**
- **Manual:** `npx synapsesync-mcp distill <session-id>`
- **Automatic:** daemon pipes captured sessions to distiller when `distill.auto: true`
- **Hook-based:** Claude Code hook fires distill after session ends

**LLM configuration:**

```bash
npx synapsesync-mcp config set distill.provider anthropic
npx synapsesync-mcp config set distill.model claude-sonnet-4-6
npx synapsesync-mcp config set distill.apiKey sk-ant-...
```

Or environment variables: `SYNAPSE_DISTILL_PROVIDER`, `SYNAPSE_DISTILL_MODEL`, `SYNAPSE_DISTILL_API_KEY`.

**Prompt design:** The distiller prompt should:
- Instruct the LLM to only extract non-trivial insights (not "user asked to fix a typo")
- Deduplicate against existing workspace content (pass existing file list for context)
- Output structured JSON with `path`, `content`, and `tags` per file
- Be provider-agnostic (works with Anthropic, OpenAI, Google APIs)

## Raw Session Lifecycle

Captured sessions are stored locally at `~/.synapse/sessions/<id>/` as JSON files. They are **not** written to the Synapse cloud workspace -- only distilled knowledge is.

- **Before distillation:** raw sessions persist locally for manual review or re-distillation
- **After distillation:** raw sessions are retained for a configurable period (default 30 days), then auto-pruned
- **Without distillation:** raw sessions persist indefinitely until manually pruned or retention kicks in
- **`capture list`** shows local sessions regardless of distillation state

## What's NOT Changing

- No new API endpoints on the backend
- No new database tables
- No new dashboard views or pages
- No changes to existing MCP server tools (`read`, `write`, `search`, etc.)
- No changes to the workspace file model
- No changes to the frontend

The capture system is entirely contained within the `synapsesync-mcp` package. It feeds into Synapse through the same `write` API that everything else uses.

## Implementation Phases

### Phase 1: Capture (ship first)
- Standard session format (`types.ts`)
- Adapter interface + 4 adapters (Claude Code, Cursor, Codex, Gemini)
- Watcher with chokidar
- Daemon with start/stop/status
- CLI commands: `capture start/stop/status/list`
- Claude Code hook integration as alternative capture path
- Tests for each adapter (unit tests with fixture files)

### Phase 2: Distill (ship second)
- Distiller with LLM prompt
- Provider abstraction (Anthropic, OpenAI, Google)
- Response parser -> markdown files
- Integration with Synapse write API
- CLI commands: `distill <session-id>`, `distill --latest`
- Auto-distill mode in daemon config
- Tests for prompt construction and response parsing

### Phase 3: Polish
- Deduplication (don't distill the same insight twice)
- Workspace-aware distillation (knows what's already in the workspace)
- Session pruning / retention policies
- Dashboard integration (optional -- show captured sessions)
